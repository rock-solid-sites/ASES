# EDASES v3

## Evidence-Driven Agentic Software Engineering System

### Mission

EDASES exists to discover, validate, and operationalize processes that allow a non-programmer principal to direct the creation of high-quality software through coordinated AI agents, measurable evidence, and layered verification.

The objective is not merely to generate software.

The objective is to create a software engineering system that:

* maximizes software quality,
* minimizes unnecessary token expenditure,
* minimizes principal oversight burden,
* preserves and compounds organizational learning,
* remains adaptable to changing model capabilities,
* increases the amount of meaningful verification available to non-programmer operators.

The system is informed by operational experience gathered across multiple completed software projects and evolves through continuous evidence collection.

---

## Core Principles

### P0: Evidence Before Architecture

Historical evidence is preferred to speculation.

Observed outcomes are preferred to theoretical arguments.

Architectural decisions should be justified by accumulated evidence whenever possible.

---

### P1: Roles Over Models

Models are interchangeable resources.

Roles are stable organizational abstractions.

Role design precedes model assignment.

---

### P2: Knowledge Must Outlive Sessions

Conversation history is not project memory.

Project memory must exist independently of any individual model, provider, or session.

---

### P3: Verification Is A Pipeline

Confidence should be derived from evidence.

Software quality should be assessed through layered verification mechanisms rather than trust in any single model, test suite, or reviewer.

---

### P4: Organizational Design Is A First-Class Concern

The primary bottleneck in mature agent systems is often coordination rather than code generation.

Task structure, information flow, role specialization, and project state are critical system components.

---

### P5: Principal Experience Matters

The system must remain operable by non-programmer principals.

Project state, risk, evidence, and decision rationale should be understandable without reading implementation details.

---

### P6: Agent-Native Engineering

Technology choices should be evaluated according to:

* agent effectiveness,
* verification potential,
* maintenance characteristics,
* long-term reliability,

rather than developer familiarity or prevailing industry convention.

---

### P7: Verification Should Be Accessible

The principal should be able to evaluate whether a system satisfies its requirements without requiring expertise in the implementation language, framework, or tooling.

Verification artifacts are preferred over implementation inspection whenever practical.

---

## Long-Term Objective

Transform software development from a process centered on implementation into a process centered on evidence, coordination, verification, and accumulated organizational learning.
